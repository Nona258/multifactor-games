# Database Schema Management Rules (Supabase / PostgreSQL)

## Schema File Detection
- Always search the project for a file named `schema.sql`.
- If `schema.sql` does not exist:
  - Create it at the project root (or the `database` directory if one already exists).
  - Add all Supabase/PostgreSQL executable SQL schema commands into this file.

## Updating Existing Schema
- If `schema.sql` already exists and contains SQL:
  - **Do NOT overwrite the entire file.**
  - Carefully update only the affected sections related to the requested schema changes.
  - Preserve unrelated tables, functions, triggers, policies, indexes, and extensions.
  - Keep the file clean, organized, and deterministic.

## Idempotent SQL Requirement (Critical)
- All SQL written to `schema.sql` **MUST** be safely re-runnable multiple times without failing.
- The schema file should execute successfully even if parts of the database already exist.

### Safe SQL Patterns
Use defensive SQL patterns such as:
- `CREATE TABLE IF NOT EXISTS`
- `CREATE INDEX IF NOT EXISTS`
- `CREATE EXTENSION IF NOT EXISTS`
- `CREATE OR REPLACE FUNCTION`
- `CREATE OR REPLACE VIEW`
- `DROP ... IF EXISTS`
- `ALTER TABLE ... ADD COLUMN IF NOT EXISTS`
- Conditional `DO $$ BEGIN ... END $$;` blocks when PostgreSQL lacks native `IF NOT EXISTS` support.

## Migration Safety
- Never assume the database is empty.
- Never create duplicate tables, policies, triggers, enums, or indexes.
- If modifying an existing table:
  - Use `ALTER TABLE` statements instead of recreating the table.
  - Preserve existing data whenever possible.
- If replacing constraints, triggers, or policies:
  - Safely drop existing versions before recreating them.

## Row Level Security (RLS)
- When adding Supabase RLS policies:
  - Check whether the policy already exists before creating it.
  - Avoid duplicate policy creation errors.
  - Ensure `ENABLE ROW LEVEL SECURITY` is only applied safely.

### Example:
```sql
ALTER TABLE profiles ENABLE ROW LEVEL SECURITY;

DO $$
BEGIN
  IF NOT EXISTS (
    SELECT 1
    FROM pg_policies
    WHERE tablename = 'profiles'
      AND policyname = 'Users can view own profile'
  ) THEN
    CREATE POLICY "Users can view own profile"
    ON profiles
    FOR SELECT
    USING (auth.uid() = id);
  END IF;
END
$$;
```

## Enum and Type Safety
- Never blindly recreate enums/types.
- Use conditional checks before creating custom types.

### Example:
```sql
DO $$
BEGIN
  IF NOT EXISTS (
    SELECT 1 FROM pg_type WHERE typname = 'user_role'
  ) THEN
    CREATE TYPE user_role AS ENUM ('admin', 'user');
  END IF;
END
$$;
```

## Output Expectations
- Always return production-safe SQL.
- Ensure the full `schema.sql` file can be executed from top to bottom repeatedly without errors.
- The schema must remain consistent, maintainable, and compatible with Supabase/PostgreSQL.
